## 1. 执行摘要

PromptSentinel 是部署在**大模型之前**的提示词安检门,实现**四道纵深防御**:构建期系统提示词加固 + canary 哨兵、输入注入检测、输出 canary 泄露检测、输出受保护标识符/PII 检测。设计原则:**纵深防御、fail-closed、数据不出域、成本可分档**。

**一句话定位**:它不是"银弹",而是大模型安全纵深防御中**专注提示词层**的一道可量化、可灰度、可持续演进的防线。

**关键成果(本系统实测)**:

| 能力 | 实测 | 说明 |
|---|---|---|
| 主线注入检测(hybrid) | **98%** | 规则 + ML(PG2)级联 |
| 主线注入检测(纯规则) | **78%** | 零依赖、亚毫秒、零成本档 |
| 中文注入检测 | **96% / F1 0.98** | 业界普遍缺口,本系统重点补强 |
| 对抗鲁棒性(归一化后) | leetspeak 58% / 间隔 32% / base64 100% | 从 **0% 完全绕过**修复至此 |
| 输出 canary 泄露 | **≈100%** | 确定性逐字检测 |
| 受保护标识符 | **100%** | 确定性 |
| 业务良性误报(全量) | **0.7%** | 贴近线上流量 |
| 评测数据集 | **9 集全威胁模型对齐** | 含清华中文集,CI 门禁守护 |
| 工程质量 | **35 单测 + CI + 回归门禁全绿** | fail-closed、优雅降级 |

## 2. 威胁模型(我们防什么,不防什么)

> 安全产品最大的不诚实是"什么都防"。我们明确边界。

**✅ 防护范围(提示词层)**:
- **提示词注入**:`忽略以上指令…`、目标劫持、指令覆盖
- **系统提示词套取**:诱导模型吐露 system prompt / 内部规则
- **越权标识符/凭证泄露**:受保护标识符、API key、PII 经输出泄漏
- **对抗混淆**:leetspeak、字符间隔、base64 编码绕过(归一化复查)

**❌ 不在防护范围(需其它手段)**:
- **内容安全 / 毒性 / 价值观**(那是内容审核的威胁模型,不是注入)——我们据此**主动剔除**了 JailbreakBench/AdvBench 等错配数据集
- **越权操作的最终执行**:须由**架构层硬边界**(最小权限、只读 RLS、egress 控制、HITL)兜底——检测层是概率性的,不可作为唯一防线

这个边界纪律贯穿全系统:连选评测数据集都以"威胁模型对齐"为第一原则,而非"越权威越好"。

## 3. 系统架构

```
                       ┌─────────────────────────────────────────┐
  用户/Agent ──TLS──▶ 网关(限流/WAF/TLS) ──▶  Guard 安检门(内网)  ──▶ 你的 LLM
                       │  ① 构建期:系统提示词加固 + canary 注入     │
                       │  ② 输入闸:规则 + ML 级联 + 去混淆归一化     │
                       │  ③ 输出闸:canary 逐字泄露检测              │
                       │  ④ 输出闸:受保护标识符 + PII/凭证          │
                       │        数据不出域:全本地推理               │
                       └─────────────────────────────────────────┘
```

**双闸设计**:输入 `screen_input` 与输出 `screen_output` 两个独立检测点,系统提示词防泄漏需两闸协同(输入拦套取意图 + 输出拦 canary/标识符)。

**成本分档**(核心工程权衡):
- **零成本档**:仅确定性规则,主线 78%、**亚毫秒**、零依赖 → 高并发/成本敏感
- **高保障档**:`use_ml_classifier`,主线 98%、**~17–25ms** → 安全优先
- **级联**(`ml_cascade`):规则命中即跳过 ML,省成本降延迟

**fail-closed**:引擎任何内部异常 → 一律拦截 + 拒绝话术,绝不静默放行。

## 4. 安全防护能力(逐防线实测)

### 4.1 九集 Benchmark(同源实测,完整指标)

每集报告 recall / precision / F1 / FPR + Wilson 95% 置信区间。最近一次评测(高保障档):

| 数据集 | 防线 | recall | F1 | FPR | 说明 |
|---|---|---|---|---|---|
| gandalf | ② 套取 | 0.85 | 0.92 | — | 系统提示词套取主线 |
| in-the-wild | ② 越狱 | 0.77 | 0.87 | — | 真实野外越狱 |
| safe-guard | ② 注入 | 0.67 | 0.80 | 0.0 | 含 1410 良性测 FPR |
| deepset | ② 注入 | 0.53 | 0.70 | 0.0 | 通用注入参照 |
| **chinese_inject** | ② 中文 | **0.96** | **0.98** | — | 清华 thu-coai,重点补强 |
| **adversarial** | ⑤ 对抗 | **0.72** | 0.84 | — | 归一化后,原本 0.5 |
| business_benign | ② 良性 | — | — | ~0(全量 0.7%) | 业务真实 FPR |
| pii | ④ 输出 | 0.38 | 0.55 | — | 输出 PII(诚实:正则口径) |

### 4.2 对抗鲁棒性(本轮重点修复)

benchmark 暴露 leetspeak / 字符间隔**完全绕过(0% 召回)**。加**输入去混淆归一化**(还原打散词 `i.g.n.o.r.e→ignore`、leetspeak `1gn0r3→ignore`、base64 解码)后复查:

| 变体 | 修复前 | 修复后 |
|---|---|---|
| leetspeak | 0% | **58%** |
| 字符间隔 | 0% | **32%** |
| base64 | 100% | 100% |
| GCG 后缀 | 82% | 82% |

零误报(4 个良性集验证)。诚实:**未达 100%**(对抗本质难),Unicode 同形字/嵌套编码仍可绕,见 §11。

### 4.3 中文安全(从盲区到 96%)

集成清华 **thu-coai/Safety-Prompts**(权威),经严格威胁模型审计——剔除其标签错配部分(Prompt_Leaking 实为良性咨询、Role_Play 为内容安全),精筛 Goal_Hijacking + 套取。叠加 6 条数据驱动的中文劫持规则(全程 train/test holdout 验证,4 个良性集零误报):**中文注入召回 ~20% → 96%**。

### 4.4 确定性防线(接近满分)

| 防线 | 实测 | 性质 |
|---|---|---|
| 输出 canary 逐字泄露 | ≈100% | 确定性(`PSENT-CANARY-` 哨兵) |
| 受保护标识符泄露 | 100% | 确定性 |
| 凭证正则(API key/PEM/AWS/GitHub/JWT…) | 高精度 | 确定性 |
| 输出 PII(姓名/地址) | 41.5% | **诚实局限**:需 NER(`use_llm_guard`) |

## 5. 评测体系(为什么这些数字可信)

可信度比数字本身更重要。本系统的评测做到:

1. **同源**:所有指标由**处理真实 `/v1/screen` 请求的同一 guard 实例**逐条现算,非预录、非另一套代码。
2. **威胁模型对齐**:9 集均对齐提示词注入,主动剔除内容安全类错配集。
3. **完整指标**:recall/precision/F1/FPR + **Wilson 95% 置信区间**(标注抽样不确定性)+ p50 延迟 + 混淆矩阵。
4. **防过拟合**:规则只在 train split 挖掘,test split holdout 验证(中文/safe-guard 增强均如此)。
5. **可复现**:seed 固定;`full=true` 全量消除抽样波动。
6. **CI 回归门禁**:`make benchmark-gate` 全量跑,召回低于下限/FPR 高于上限即 **exit 1**,接入 CI——改规则导致退化立即变红。
7. **持续演进**:评测结果持久化(带 run_id/时间戳),`/api/benchmark/history` 可查趋势。

> 一个有代表性的诚实案例:benchmark 曾显示业务 FPR 12.5%,深查发现是 **n=40 抽样噪声**,全量实为 0.7%。这正是"全量 + 置信区间"方法学的价值。详见 `docs/BENCHMARK-METHODOLOGY.md`。

### 6.1 完整度审查(多 agent 交叉,已修复)

经**配置文件 / 容器镜像 / 代码接线**三维度并行审查(每条发现独立验证真伪),确认 13 项、**修复 6 项**(1 高 + 2 中 + 3 重要低):
- **高**:影子模式与 benchmark 交叉污染——评测走 `screen_input` 会被 `_shadow` 改写,shadow 部署下 recall 塌 0。修复:评测加 `_apply_mode=False` 旁路,永远按 enforce 判定(实测 shadow 下:修复后 0.74 vs 旧路径 0.00)。
- **中**:`/metrics` 加 `mode` 标签消除 enforce 下的死指标;Dockerfile 拆 `requirements-pg2.txt` 单一事实源 + 权重预下载断言(fail-fast)。
- **低**:`validate_config` 校验 `server.mode`(防影子拼错→静默误拦)、对抗变体长度封顶、telemetry_reset 保留 mode。
- 余 7 项低危为信息性/可移植性/设计取舍,已确认非 bug,记录留待优化。

价值在于**抓到了开发者自己引入的跨功能副作用**:影子模式与 benchmark 单独都对,交叉处才暴露——这正是多视角交叉审查不可替代之处。修复后回归 35 绿、门禁 PASSED、容器健康。

## 7. 生产部署与运维

| 能力 | 实现 | 状态 |
|---|---|---|
| 容器化 | 非 root(uid 10001)、健康检查、镜像预装 ML 权重 | ✅ |
| 网络隔离 | Guard 仅内网可达,不绑宿主机端口 | ✅ |
| 速率限制 | 进程内 240/min/IP(`SENTINEL_RATE_PER_MIN` 可调) | ✅ + 建议叠加网关层 |
| 安全响应头 | `X-Content-Type-Options/X-Frame-Options/Referrer-Policy` | ✅ 实测生效 |
| 资源保护 | 输入 20000 字符上限(防 ReDoS/OOM/成本爆炸) | ✅ 实测 30KB 不崩溃 |
| 鉴权 | `server.auth_token`(Bearer) | ✅ 生产必填 |
| **影子模式** | `mode: shadow` —— 本会拦的改放行 + 标记 `would_block` | ✅ **灰度上生产标准第一步** |
| 监控遥测 | 只记 reasons/risk/耗时,**绝不记 prompt/response 正文** | ✅ |

**影子模式(灰度上生产的关键)**:设 `mode: shadow`,引擎照常检测但**不实际拦截**,仅在响应标记 `would_block`。可在生产真实流量上观测"如果开拦截会拦多少、误拦多少",**不影响业务**;确认 FPR 可接受后再切 `enforce`。这是把概率性检测器安全推上生产的工程化路径。

## 8. 接入方式

- **4 语言 SDK**:Python / JavaScript / Java / Go,+ 纯 HTTP(任意语言)
- **五步接入**:构建加固系统提示词 → 注入 canary → 输入 `screen_input` → 调用你的 LLM → 输出 `screen_output`
- **Web 门户**:能力介绍 / 实时 demo / 接入 playground(四语言代码生成 + 在线试调)/ 真实 benchmark / 监控遥测
- **零改造接入**:作为大模型前的旁路安检门,不改你的 LLM 与业务逻辑

---

## 9. 数据安全与合规

- **数据不出域**:全本地推理(规则 + 本地 ONNX),无外呼。`use_llm_judge`(会外呼)默认关闭,开启且指向外部时 selfcheck 告警。
- **不留正文**:日志与遥测只记判定元数据(reasons/risk/耗时),从不持久化 prompt/response 正文。
- **最小权限**:容器非 root,网络内网隔离。

## 11. 诚实局限与上生产路线图

**已知局限(不藏)**:
1. **对抗未达 100%**:Unicode 同形字、嵌套编码可绕 → 建议叠加归一化网关 + 高风险人审。
2. **中文靠 regex**:ML 对中文弱(96% 来自规则),随新表达演进需持续补 —— 但有 benchmark + 门禁守护。
3. **输出 PII 41.5%**:姓名/地址需开 `use_llm_guard`(NER)。
4. **检测是概率性的**:越权/外泄须架构层硬边界兜底。
5. **缺真实验证**:压测/渗透/私有数据需在接入方环境补齐。

**上生产路线图(建议按序)**:
1. **影子模式灰度**(已就绪):`mode: shadow` 跑真实流量,观测 `would_block` 率与误拦,**不影响业务**。
2. **用真实流量定 FPR**:用接入方业务样本替换 `business_benign`,得到线上真实误报率。
3. **导入私有红队集**:防公开集污染,持续补 benchmark 暴露的漏报。
4. **压测 + 渗透**:QPS/p99/资源 + 安全审计。
5. **切 enforce + 多实例 + 网关限流 + 监控告警**:正式承载。

## 13. 结论

PromptSentinel 在**提示词安全这一专门威胁模型**上,提供了**可量化(9 集对齐 benchmark + 完整指标)、可灰度(影子模式)、可持续演进(CI 门禁 + 私有集机制)、诚实可信(威胁模型边界清晰 + 局限不藏)**的工程实现。

**给接入方的核心建议**:从**影子模式灰度**起步 —— 零业务风险地用你的真实流量验证它,数据会自己说话。
