# 团队接入指南(PromptSentinel)

> 本工程是**标准模板**:你复制一份,改一个配置文件,自检,部署。
> **各团队各自部署自己的容器**,互不影响。

## 你只需要改一个文件

整个工程里,你唯一要动的是 **`sentinel.config.yaml`**。`app/` 下的代码是安全核心,**不要改**
(改了会破坏统一安全基线)。

| 文件/目录 | 能不能改 | 说明 |
|---|---|---|
| `sentinel.config.yaml` | ✅ **改这里** | 你的受保护标识符、拒绝话术、阈值、扫描器开关 |
| `app/engine.py` `app/patterns.py` | ⛔ 不改 | 四道防线核心逻辑与规则 |
| `app/main.py` `app/config.py` | ⛔ 不改 | 服务与配置加载 |
| `Dockerfile` / `docker-compose.yml` | ⚠️ 一般不改 | 端口冲突时可改映射端口 |

## 五步上手

### 第 1 步 · 复制工程
把 `service/` 整个复制到你团队的仓库(或 fork)。

### 第 2 步 · 改配置 `sentinel.config.yaml`
最少要改两处:
```yaml
team:
  name: "你的团队名"          # 如 supply-chain
  agent: "你的 Agent 名"

protected_terms:              # 填你自己的受保护标识符
  - "你的 Action 编号"
  - "你的对象/工具/schema 名"
```
其余(拒绝话术、阈值、扫描器开关)按需调,默认值即可用。

### 第 3 步 · 自检
```bash
pip install -r requirements.txt
SENTINEL_ALLOW_DEFAULT=1 python -m app.selfcheck
```
看到「自检通过…可以部署」才继续。有 FAIL 按提示修;有 WARN 建议处理
(比如 team.name 还是示例值、protected_terms 为空)。

### 第 4 步 · 部署你自己的容器
```bash
docker compose up -d
curl http://localhost:8000/health
# {"status":"ok","team":"你的团队名",...}
```
> 改了 `sentinel.config.yaml` 后,`docker compose restart` 即生效(配置是挂载的,不用重建镜像)。

### 第 5 步 · 接到你的 Agent 链路
三步调用(任何语言 HTTP 调用即可,业务端无需 Python;或用 `../sdks/<语言>` 的客户端 SDK):
```
①部署时   POST /v1/system-prompt/build  → 拿 hardened_system_prompt + canary(存好)
②请求时   POST /v1/screen/input         → allowed=false 则返回 refusal,不调模型
          调用你的大模型(system=hardened_system_prompt)
③返回前   POST /v1/screen/output(带 canary)→ 返回 text(不通过时已是拒绝话术)
```
详细示例见 `README.md` 与 `../sdks/`(Python/JS/Go/Java)、`../docs/INTEGRATION.md`。

## 接入合格标准(自检 + 红队)

- `SENTINEL_ALLOW_DEFAULT=1 python -m app.selfcheck` 全 PASS、无遗留 WARN。
- `python -m app.e2e_demo` 能看到四道防线分别拦住对应攻击(含 canary 出口兜底)。
- `SENTINEL_ALLOW_DEFAULT=1 python -m benchmark.run_benchmark`:确定性层 recall≥0.9、fpr=0。
- **上生产前**:用你团队**真实的攻击样本**补充 `redteam/corpus.py`,关注误报率(过度拦截比漏拦更常见),
  按结果微调 `thresholds` 与 `protected_terms`;必要时跑 `python -m redteam.llm_redteam` 做 LLM 驱动探测。

## 重要边界(务必知道)

本服务是纵深防御的**提示词层 + 检测层**,概率性、可被绕过。**它不替代架构硬边界**:
最小权限、工具凭证不进模型、只读/RLS、egress 管控、高危动作 HITL —— 这些必须由你的平台/Runtime 实现。
详见 `../docs/SECURITY-BOUNDARIES.md`。

## 常见问题

- **端口被占用?** 改 `docker-compose.yml` 里的 `"8000:8000"` 左边端口。
- **想要更强的通用注入/PII 检测?** 设 `use_llm_guard: true` 并装 `requirements-ml.txt`(镜像更大、首启下载模型,仍离线运行、数据不出域)。
- **能不能用 LLM 做语义裁决?** 可设 `use_llm_judge: true`,但⚠️ 若 `llm_judge.base_url` 指向**外部 API**,
  会把待检测文本发往外部,**破坏数据不出域**——自检会告警;仅在指向**自托管**模型时启用。
- **误报太多?** 调高 `thresholds.input`(如 0.5→0.65),或精简 `protected_terms`。
- **漏报?** 调低阈值,或补 `protected_patterns`/`protected_terms`,或启用 ML/LLM-judge,并以架构层兜底。
